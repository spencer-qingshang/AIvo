在《交换与路由技术》的考试和实际工程中，“设备间通信状态检验”俗称**网络排错（Troubleshooting）**。当两台设备“失联”时，你不能瞎猜，必须按照网络的分层逻辑，掏出网络工程师随身携带的“三把手术刀”。

### 第一把手术刀：网络听诊器 —— `ping` 命令

这是测试网络连通性最经典、最高频的指令，它基于网络层的 **ICMP 协议**（互联网控制报文协议）。

- **工作原理：** 发送端扔过去一个 `Echo Request`（回显请求）报文，如果对方活着且网络通畅，就会立刻扔回来一个 `Echo Reply`（回显应答）报文。
    
- **考场高频参数（华为 VRP）：**
    
    - 普通 Ping：`ping 10.1.1.1`
        
    - **带源 Ping（大题必考）：** `ping -a 192.168.1.1 10.1.1.1`。_（为什么要带源？因为路由器有多个接口，如果不指定源 IP，路由器会默认用离目标最近的出接口 IP 去 Ping，这无法真实测试内部主机到外部的连通性。）_
        
- **如何看懂“体检报告”（三种经典回显）：**
    
    1. `Reply from 10.1.1.1: bytes=56 Sequence=1 ttl=254 time=10 ms` ➡️ **全通！** 完美。
        
    2. `Request time out` ➡️ **请求超时。** 包发出去了，但没回来。可能是对方关机了、中间路由器丢包了，或者被对方防火墙的安全策略（拒绝 ICMP）给无情拦截了。
        
    3. `Destination host unreachable` ➡️ **目标不可达。** 连怎么去目标的“路（路由）”都找不到，直接死在半道上了。
        

### 第二把手术刀：网络 X 光机 —— `tracert` (Traceroute) 命令

当 Ping 不通（或者 Ping 延迟极高）时，你需要知道数据包到底死在了哪条路上。这时候必须拔出比 Ping 更强大的武器：路径追踪。

- **工作原理（极其巧妙的设计）：** 它利用了 IP 报文里的 **TTL（生存时间）** 字段。
    
    1. 源主机先发一个 TTL=1 的探测包，包刚走到**第一跳（网关）**，TTL 耗尽，网关丢弃该包并回传一个“超时报错”。这样你就知道了第一跳的 IP。
        
    2. 接着发 TTL=2 的包，走到**第二跳**报错，你就知道了第二跳的 IP。
        
    3. 以此类推，直到抵达最终目的地。
        
- **实战口令：** `tracert 10.1.1.1`
    
- **排错绝招：** 看着屏幕上一行行打印出来的 IP，**停在哪一个 IP 不动了（出现星号 `* * *`），就说明故障出在哪一台路由器上**。直接登录那台设备查路由表或查 ACL 拦截！
    

### 第三把手术刀：底层状态扫描仪 —— `display` 命令族

在考场的大题排错中，如果网络不通，一定要按照**“自底向上（从物理层到网络层）”**的顺序，用 `display` 命令给设备做全身扫描。

**1. 查物理层与数据链路层（看门有没有开）：**

- **命令：** `display interface brief` （查看接口简要状态）
    
- **得分眼：** 必须看到接口的物理状态（PHY）和协议状态（Protocol）都是 **`UP / UP`**。
    
    - 如果是 `DOWN / DOWN`：网线没插、或者被管理员手动 `shutdown` 了。
        
    - 如果是 `UP / DOWN`：物理线插好了，但链路层协议没协商好（比如昨晚刚复习的 PPP CHAP 密码配错了！）。
        

**2. 查网络层（看有没有地图）：**

- **命令：** `display ip routing-table` （查看路由表）
    
- **得分眼：** 数据包要能过去，还得能回来。如果在路由表里根本找不到去往目标网段的条目，说明 OSPF 没配对或者静态路由写漏了，直接补路由。

### 🎯 考场排错“肌肉记忆”流程图

遇到“PC 1 无法访问 Server 1，请排错”这种综合大题，脑子里直接跑这段程序：

1. **先敲 `ping`：** 确认是真的不通，还是只是慢。
    
2. **不通？敲 `tracert`：** 查出数据包具体死在了哪个接口、哪台设备上。
    
3. **锁定设备后，敲 `display interface`：** 看是不是接口 Down 了，或者接口 IP 敲错了、掩码写错了。
    
4. **接口没问题？敲 `display ip routing-table`：** 查路由表，看这台设备知不知道去目标的路怎么走。
    
5. **路由也有？查防火墙 / ACL：** 看是不是昨晚复习的那些安全策略、高级 ACL 把流量给“误杀”了。
