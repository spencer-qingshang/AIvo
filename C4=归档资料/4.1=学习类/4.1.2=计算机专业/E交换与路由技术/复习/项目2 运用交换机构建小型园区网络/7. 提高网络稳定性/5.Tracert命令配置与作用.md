在备考冲刺阶段，如果说 `ping` 命令是用来判断网络“通不通”的听诊器，那么 **`tracert`（路由追踪）** 就是用来精确定位网络到底“死在哪一跳”的 X 光机。

在企业网络排错和《交换与路由技术》的综合大题中，`tracert` 是极其关键的排障手段。

### 1. Tracert 的核心作用

当你用 `ping` 测试一个跨越了多台路由器的目标 IP 时，如果结果是 Request timeout（请求超时），你只能知道网络不通，但几百公里的线路上，到底是哪台路由器配置错了？

这时候输入 `tracert 目标IP`，它会：

- **逐跳显示路径**：把你的数据包从源端到目的端经过的**每一台路由器（每一跳）的 IP 地址**全部按顺序打印出来。
    
- **精确定位断点**：如果数据包在经过 R 1、R 2 后，在 R 3 处迷路了（比如 R 3 没有配置到达目的地的路由表），那么 `tracert` 的结果就会在打印完 R 2 的 IP 后，开始出现一排排的 `* * *`（超时）。你立刻就能断定：“故障就在 R 2 到 R 3 之间，我要去检查 R 2 的路由配置！”
    

### 2. VRP 平台下的 Tracert 配置与命令

`tracert` 不需要像协议那样进行复杂的系统配置，它是一个直接在命令行执行的测试工具，**在用户视图 `< >` 或系统视图 `[ ]` 下均可使用**。

**基础用法：**

Plaintext

```
<Huawei> tracert 192.168.2.10
```

_执行后，屏幕会逐行输出到达 `192.168.2.10` 所经过的路由器接口 IP 及延迟时间。_

**进阶用法（考试排错必杀技）：**

路由器通常有多个接口，默认情况下 `tracert` 会用离目标最近的那个出接口 IP 作为源地址。但在复杂的 OSPF 或静态路由实验中，我们必须**指定源地址**来模拟特定网段的通信路径。

- **带源地址的追踪 (`-a` 参数)**：
    
    Plaintext
    
    ```
    <Huawei> tracert -a 192.168.1.254 192.168.2.10
    ```
    
    _作用：强制指定用 `192.168.1.254` 这个 IP 去追踪到达目标的路径，这在排查单向路由故障时极其有效。_
    

### 3. 底层原理：它是怎么实现“逐跳记录”的？（选择题高频考点）

`tracert` 的魔法来自于 IP 报文头部的一个关键控制字段：**TTL（Time To Live，生存时间）**。

TTL 的作用是防止数据包在网络里出现环路而永远“无限循环”。数据包每经过一台路由器，TTL 的值就会减 1。当 TTL 减到 0 时，路由器就会把这个包丢弃，并给源主机回传一个 **ICMP 超时（Time Exceeded）** 报错信息。

`tracert` 巧妙地利用了这个机制：

1. 它先发出一个 **TTL = 1** 的测试包。包到了**第一跳**路由器，TTL 减为 0，第一跳路由器丢弃包并报错。主机收到报错，顺利记录下第一跳路由器的 IP。
    
2. 它接着发一个 **TTL = 2** 的包。包安全通过第一跳，到了**第二跳**路由器，TTL 减为 0，第二跳路由器报错。主机记录下第二跳路由器的 IP。
    
3. 如此不断递增 TTL（3, 4, 5...），直到数据包最终到达目的地。目标主机收到包后，会返回一个端口不可达或正常的响应，主机就知道追踪完成了。
