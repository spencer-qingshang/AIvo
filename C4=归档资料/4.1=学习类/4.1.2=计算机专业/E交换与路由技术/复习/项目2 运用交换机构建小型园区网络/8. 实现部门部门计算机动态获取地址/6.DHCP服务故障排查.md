面对马上就要到来的 4 月份《交换与路由技术》考试，如果在 eNSP 的综合大题里，你把代码敲完，满心欢喜地打开 PC 点下“自动获取”，结果却跳出一个 `169.254.x.x` 的无效地址（或者一直在获取中），这种瞬间的窒息感是每个网络工程师都经历过的。

DHCP 故障排查（Troubleshooting）不仅是考卷上的高频排错题，更是检验你是否真正打通了二层 VLAN 和三层 IP 逻辑的试金石。

排查 DHCP 故障，千万不要像无头苍蝇一样乱敲命令。我们需要遵循**“从底向上，从端到端”**的排错逻辑：

### 第一阶段：底层与链路检查（最容易被忽视的低级错误）

很多时候 DHCP 拿不到地址，根本不是 DHCP 的代码配错了，而是底层的“路”不通，导致 PC 发出的 `DHCP Discover` 广播包根本到达不了网关。

1. **检查端口与 VLAN**：
    
    - PC 连接的交换机端口是不是亮绿灯（UP 状态）？
        
    - **核心考点**：该端口是否正确划分到了对应的 VLAN？（比如 PC 在 VLAN 10，结果端口默认还在 VLAN 1，广播包直接走岔了）。
        
    - _排错命令_：在接入交换机上敲 `display port vlan` 确认。
        

### 第二阶段：网关与 DHCP 服务侧检查（考场核心丢分区）

如果二层路通了，接下来就要检查路由器或三层交换机上的配置了。这里隐藏着三个极其经典的“致命陷阱”：

1. **陷阱一：全局开关没开**
    
    - 这是 eNSP 考试中最惨的死法。你在地址池里配得天花乱坠，但只要漏了全局视图下的那句 **`dhcp enable`**，设备就根本不处理任何 DHCP 报文。
        
2. **陷阱二：接口忘做绑定**
    
    - 地址池（Pool）建好了，接口（如 Vlanif 10）的 IP 也配好了，但你没有在接口下告诉它怎么发牌。
        
    - _补救命令_：必须进接口敲 **`dhcp select global`**（去全局池拿）或者 **`dhcp select interface`**（就地拿）。
        
3. **陷阱三：网关 IP 与地址池不在同一网段**
    
    - 如果 Vlanif 10 的 IP 是 `192.168.10.254/24`，但你的地址池 `network` 却宣告成了 `192.168.20.0 mask 24`，接口在收到请求时，会发现地址池跟自己不在一个频道，直接拒绝分配。
        

### 第三阶段：核查地址池状态（服务端深度诊断）

如果配置全对，但还是拿不到 IP，我们需要看看“池子”里到底还有没有水。

- **排错神令：`display ip pool`**
    
    敲下这行命令，看输出的统计信息：
    
    - **Idle（空闲）**：如果为 0，说明地址全部分光了。你需要扩大子网掩码，或者缩短租期（lease）。
        
    - **Conflict（冲突）**：如果有大量冲突，说明局域网里有人私自设了静态 IP 抢占了资源，这时候就要结合我们上一节讲的 ARP 知识去抓人了。
        
    - **Excluded（排除）**：检查是不是不小心把整个网段都 `excluded-ip-address`（排除分配）了。
        

### 第四阶段：跨网段中继排错（DHCP Relay 专属）

如果题目是分离的 DHCP 服务器架构（比如中间隔着核心路由器），你需要额外排查两点：

1. **中继配置对了吗？** 终端的网关接口下有没有敲 `dhcp select relay` 和 `dhcp relay server-ip [真实服务器IP]`。
    
2. **路由通了吗？** 这是最关键的！在网关上用 `ping` 测试一下那个真实的 DHCP 服务器 IP。如果网关自己都 ping 不通 DHCP 服务器，那它拿什么去帮你“代购” IP 呢？（这通常意味着你需要补一条静态路由或 OSPF）。
    

---

**考场急救三板斧总结：**

1. 查全局 `dhcp enable` 敲了没。
    
2. 查接口 `dhcp select ...` 敲了没。
    
3. 查接口 IP 网段和地址池网段对不对得上。
    

打通了 DHCP，我们就彻底完成了局域网内部终端的接入和互通。从 VLAN 划分到网关冗余（VRRP），再到 IP 自动分配（DHCP），你已经啃下了《交换与路由技术》二层和边界网络的大半江山。
