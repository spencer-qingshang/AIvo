当一个数据包（报文）来到路由器的接口时，ACL 规则的真实匹配流程分为以下几个决定生死的核心关卡：

### 第一关：判定调用方向（Inbound vs Outbound）🌟最易错考点

ACL 规则写得再好，如果没有挂载到接口的正确方向上，也等于废纸一张。路由器首先会看数据包是“进”还是“出”。

- **Inbound（入方向）：** 数据包**刚刚抵达**路由器的接口，还没进入路由器的内部进行查表转发。
    
    - _处理逻辑：_ 路由器先用 ACL 检查它，**如果被 Deny（拒绝），直接在接口外丢弃**，既节省了路由器的 CPU 算力，又节约了内部带宽。这也是安全过滤最推荐的方向。
        
- **Outbound（出方向）：** 数据包已经穿过了路由器内部，查完了路由表，**正准备从接口发出去**。
    
    - _处理逻辑：_ 路由器在最后关卡用 ACL 检查它。如果被 Deny，数据包在出站的最后一刻被丢弃。
        

### 第二关：提取报文特征并激活 ACL

如果接口上确实挂载了 ACL，路由器会像安检 X 光机一样，提取数据包的特征（源 IP、目的 IP、端口号等），并调出对应的 ACL 列表。

### 第三关：进入“命中即停”的循环比对（核心逻辑流）

这是 ACL 匹配的心脏地带，严格遵循我们之前提到的原则：

1. **取第一条规则（Rule）：** 按照设定好的顺序（Config 按编号，Auto 按深度），抽出优先级最高的第一条规则。
    
2. **比对特征：**
    
    - **如果不匹配：** 路由器什么都不做，立刻**抽取下一条规则**继续比对。
        
    - **如果匹配（命中！）：** 路由器立刻看这条规则的动作是 Permit 还是 Deny。
        
3. **执行动作并终止：**
    
    - 如果是 **Permit（允许）**：立刻放行该数据包，**并且彻底停止查阅后面的所有 ACL 规则**。
        
    - 如果是 **Deny（拒绝）**：立刻丢弃该数据包，**同样彻底停止查阅后面的规则**。
        

### 第四关：查无此人的“兜底动作”（隐式规则）

如果在循环比对中，数据包把所有的 Rule 都对了一遍，发现**连一条都没命中**，这时就会触发系统的默认机制：

- 在华为设备的接口包过滤（Traffic-filter）中，默认兜底动作是 **Permit（放行）**。
    
- 在 Telnet/SSH 等远程访问控制中，默认兜底动作是 **Deny（拦截）**。

### 🎯 考场实战沙盘演练

**考题情景：** 路由器接口 `GigabitEthernet0/0/1` 的 **Inbound** 方向配置了如下 ACL：

Plaintext

```
rule 5 deny source 192.168.1.10 0
rule 10 permit source 192.168.1.0 0.0.0.255
```

**问题：** 此时，主机 `192.168.1.10` 和 `192.168.1.20` 分别向该接口发送数据包，命运如何？

**秒杀推演：**

1. `192.168.1.10` 的数据包抵达：
    
    - 看 Rule 5：精准命中！动作是 Deny。
        
    - **结果：立刻丢弃，不看 Rule 10。**
        
2. `192.168.1.20` 的数据包抵达：
    
    - 看 Rule 5：不匹配（IP 不同），继续往下走。
        
    - 看 Rule 10：属于 `192.168.1.0/24` 网段，命中！动作是 Permit。
        
    - **结果：成功放行。**
